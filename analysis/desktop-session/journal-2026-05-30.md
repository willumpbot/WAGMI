**Trader's read:** On HYPE and SOL, the LLM's caution is well-calibrated — every skipped trade would have lost. On ETH, every single skip would have won. That's not random. The bot has a calibration drift specifically on ETH.

**Caveat:** Small sample (6 ETH skips), measurement window is short (most signals haven't had enough time to fully play out at TP1/SL targets that take many hours). Six identical outcomes IS unusual though — coin-flip probability is ~1.5%.

## Finding 3 — Regime-label mismatch: ETH boost rule never fires

**The rule** (`eth_trending_regime_boost_v1`):
```json
"conditions": {"symbol": "ETH", "regime": "trend"},
"action": "boost", "adjustment": 8.0,
"hypothesis": "ETH in TRENDING regime = 71% WR, +$9.49/trade EV on 7 live trades."
"times_applied": 0
```

**Why it never fires:** The condition checks for `regime: 'trend'`. But the live bot's regime detector reports ETH as `'trending_bull'` when ADX is high (saw this in the bot_thinking dashboard live). The labels don't match.

**Trader's read:** We have a rule that says "boost ETH when trending" — exactly what should fire on the trades the bot just missed. It never fires because of a string-label mismatch. The hurtful penalize/veto rules don't have this issue (they match on symbol+side, not regime).

**Bias direction:** The rule system is asymmetric — the harmful rules fire constantly, the helpful boost rule for the symbol we're missing wins on (ETH) never fires.

## Finding 6 — LLM agents have a "always skip" lock-in

**What I checked:** All 55 agent_performance.jsonl decision records.

**Per-agent decision distribution:**

| Agent | Calls | Decisions | Avg Conf |
|---|---|---|---|
| Trade | 7 | 7 skip, 0 go, 0 flip | **0.24** |
| Critic | 7 | 7 approve | 0.50 |
| Regime | 7 | 6 consolidation, 1 range | 0.73 |
| Risk | 7 | 7 size=0,override=skip | 0.50 |
| Quant | 7 | 7 "unknown" (output broken) | 0.50 |
| Scout | 20 | 20 monitor | 0.50 |

**Critical observations:**

1. **The Trade agent is in "always skip" mode** with very low confidence (0.24). It hasn't recommended a single trade today.
2. **Critic approves 100% of the time** — but 7/7 of those were approving Trade's "skip" decision. The Critic has never been faced with a "go" decision to actually critique. The contrarian-check function isn't being exercised.
3. **Regime agent disagrees with mechanical regime detector.** Agent says `consolidation/range`. Mechanical detector says `trend/trending_bull`. Both are seeing the same data, different timescale interpretations. Price moved 0.06% in 3h which IS consolidation behavior — the LLM agent may actually be more correct here.
4. **Quant agent returns "unknown" 7/7 times** — the parser isn't extracting a valid decision label. Either prompt issue or parser issue.
5. **Scout: 20 calls, all "monitor"** with confidence 0.50 — never escalates beyond passive watching.

**Trader's read:**

The bot is in a state where every agent's output is at minimum-confidence baseline (~0.50) except Trade (0.24, even lower) and Regime (0.73 confident in consolidation). The system has effectively reached "agree to wait" consensus.

This is internally consistent: if the market really is consolidating and Trade has low conviction, then skip + critic-approve is correct. But the Critic never seeing a "go" decision means we don't know if it would actually disagree — that part of the system is untested.

**Three diagnostic options when Nunu wants to push for trades:**

1. **Loosen Trade agent confidence threshold** — current Trade output has avg conf 0.24 which doesn't pass any reasonable bar. If we want the Trade agent to actually take shots, its prompt needs tuning to be less hedging.
2. **Fix the Regime agent vs Regime detector mismatch** — pick one classifier or have one explicitly inform the other. Right now they disagree silently.
3. **Fix Quant parser** so its output is real (currently always "unknown").

## Intervention 1 — 4 changes to push the bot to actually trade

Per Nunu's authorization at 2026-05-30 ~18:30 PT. The bot has run 9 hours, 0 trades, 257 skips. Each missed signal is data lost. Making 4 surgical changes:

### Change 1: ENSEMBLE_CONFIDENCE_FLOOR lowered from 55 → 20

In `bot/.env`. Per Nunu's recall of a prior backtest showing 20% confidence threshold was optimal — "we can truly read more signals." Setting the floor to 20 lets the LLM see signals that were previously gated before the multi-agent pipeline ever ran.

### Change 2: Disabled `night_session_block_v1` rule

In `bot/data/llm/graduated_rules.json`. This rule had 344 applications, 0 outcome validations (broken feedback loop). Hard veto on 00:00-06:00 UTC trades based on unvalidated historical data. With overdrive mode, we want to collect data from ALL sessions — if night sessions truly underperform, the LLM will discover it from real outcomes.

### Change 3: Disabled `conf_floor_70_v1` rule

Same file. This rule penalized 60-70% confidence signals by -20 points based on stale "46.3% WR on 123 trades" historical evidence. 105 applications, 0 validations. Directly contradicts Nunu's prior "20% confidence is optimal" backtest finding. Likely the largest single contributor to the always-skip pattern.

### Change 4: Fixed `eth_trending_regime_boost_v1` regime label

Changed condition from `regime: "trend"` to `regime: "trending_bull"`. The mechanical regime detector emits `trending_bull/bear` not `trend`, so this rule never fired despite ETH being in trending_bull constantly. This is the root-cause fix for the ETH "all missed wins" pattern from Finding 1.

### Change 5: Added OVERDRIVE MODE preamble to TRADE_AGENT_PROMPT

In `bot/llm/agents/prompts.py`. Six-line block at the top explicitly stating:
- This is paper trading; goal is trade outcomes for learning
- Default to "go" when thesis is plausible + regime supports + no clear contraindication
- Reserve "skip" for clear negative evidence (not ambiguity)
- Solo signals with coherent thesis are tradeable
- Don't compensate for the lowered confidence floor by being more skeptical
- If skipping 5+ in a row, you're over-filtering

This is the most direct push against the always-skip pattern: tells the Trade agent explicitly that the bias should be toward action, with skip as the exception.

## Finding 8 — Pipeline-level gates were the final blocker (2026-05-31)

After 9+ interventions on agent prompts and rule disabling, the bot still wasn't trading. Investigation found `signal_pipeline.py:339` has an EV gate that hard-rejects any signal with `ev_per_dollar < min_signal_ev`. Default was 0.08. **Every signal we saw had EV in the range -0.21 to -2.0**, meaning ALL of them would be rejected here even after the agent pipeline approved.

This gate is NOT governed by `LLM_FIRST_MODE`. It runs unconditionally on every signal post-agents. So even with all the agent prompts tuned to OVERDRIVE, the live bot still rejected everything at this layer.

**Fix**: Added to `.env`:
- `MIN_SIGNAL_EV=-3.0` (was 0.08)
- `MIN_SIGNAL_RR=1.0` (was 1.2)
- `MIN_STOP_WIDTH_PCT=0.002` (was 0.004)

These allow significantly negative-EV trades through, consistent with Nunu's "collect trade data even at the cost of paper losses" directive for overdrive mode.

Bot restarted PID 18592.

## Finding 9 — Quant agent "unknown" was a parser bug, not the agent's fault

Fixed in commit `089940a`. Quant decisions will now log as `"long,kelly=0.50,noise=0.78"` instead of `"unknown"`. This matters because Trade Agent reading downstream sees Quant's decision label and may have been weighting "unknown" as low-conviction skip signal.

## Finding 10 — Graduated rules file kept getting overwritten

When I set `night_session_block_v1` and `conf_floor_70_v1` to `active: false`, the bot's running process had the old `active: true` in memory (loaded at startup). On the next `evaluate_signal()` call, `ensemble.py:1006` -> `_save()` -> wrote in-memory state to disk, overwriting my edit.

This is the "two writers, one file" race condition. To make rule changes stick, the bot MUST be restarted between the edit and the next signal eval cycle. After multiple restarts the rules now stay disabled.

Going forward: any rule edits should be paired with an immediate bot restart, and verified again 5+ minutes later.

## Entry 12 — 2026-05-31 04:01 UTC — The Breakthrough: Hidden `<60` LLM Bypass Gate

### Context
After 11 prior interventions (lowered floor, made gates informational, added shadow edges, fixed Quant parser, etc.), the bot still took 0 trades through the overnight period and the LLM agents weren't even being invoked despite signals flowing through. Subscription just reset, gave me a fresh shot at finding why.

### Reading the log differently
Previous traces all stopped at "QUANT-BRAIN ETH BUY → go" and I assumed the LLM pipeline ran silently after. Wrong. Looking at the *next* log lines after each Quant approval revealed: NO multi-agent pipeline call. NO Trade Agent. NO Risk Agent. NO Critic. Just the next 30-second scan.

The Multi-Agent coordinator only logged Scout and Overseer — never Trade/Risk/Critic. The bot wasn't asking the LLM about ETH at all, even after Quant approved.

### Finding 1: Adaptive floor overriding `.env` every scan
`bot/feedback/adaptive_confidence.py` has hardcoded constants:
- `ABSOLUTE_MIN_FLOOR = 50.0`
- `DEFAULT_FLOOR = 55.0`

`multi_strategy_main.py:1727` reads `confidence_floor.current_floor` and writes it to `self.ensemble.confidence_floor` every scan. So even though `.env` says `ENSEMBLE_CONFIDENCE_FLOOR=20`, the adaptive module overwrites it to 55 every cycle. Log evidence: `[ADAPTIVE-FLOOR] Updated ensemble confidence floor from 20.0 to 55.0`.

**Fix applied:** When `LLM_FIRST_MODE=true`, cap the adaptive floor at the configured value. Adaptive can lower but cannot raise above the user-set baseline.

### Finding 2 (the actual killer): Hardcoded `<60` cost gate
`multi_strategy_main.py:4530` (now-removed code):
```python
if _sig_conf < 60:
    logger.info("LLM SKIP: confidence X% < 60% threshold")
    _llm_first = False
    _llm_dual_track = False
```

This is *before* the LLM-first path executes. Every signal under 60% confidence was force-routed to the mechanical path — which has the EV gate that's been rejecting everything for hours. ETH BUY at 52% never had a chance: the Quant Brain (rule-based) approved it, but the *actual LLM agents* never saw it because this cost gate bypassed the entire LLM-first dispatcher.

**Fix applied:** `_llm_first_min = min(60.0, float(self.config.ensemble_confidence_floor))`. With `ENSEMBLE_CONFIDENCE_FLOOR=20` from .env, this is now 20, so ETH at 52% enters the LLM-first path.

### Why I missed this for 11 hours
The trace I was looking at went:
1. Raw signal generated
2. Ensemble votes
3. EV-rejection made informational  
4. → forwarded to LLM
5. Quant-Brain → go
6. ... nothing.

But step 4 ("forwarded to LLM") was about a *different* LLM path (the ensemble's monolithic pre-LLM stage). The LLM-first multi-agent dispatcher is a separate code path at line 4538 that requires `_llm_first = True`. That flag had been set to `False` at line 4536 by the cost gate — invisible in the log because the SKIP message uses `logger.info` and gets buried in the noise.

### Commit
`ed330de` on `desktop-overdrive-2026-05-30`. Bot restarted at PID 18196 to apply.

- Any new "Feedback floor BLOCKED" lines (separate gate that lives in mechanical path; should be bypassed by LLM-first).

### Risk Agent + leverage.py audited — no other silent killers

- `execution/leverage.py:107` has `if confidence < 20: return no-trade`. We're at floor=20, so ETH-class signals at 50%+ comfortably pass.
